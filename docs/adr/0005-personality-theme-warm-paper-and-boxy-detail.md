# Replace the single generic accent with a palette and detail layer grounded in named references

`CODING_STANDARDS.md:280` previously documented templui's default blue as a deliberate but arbitrary choice ("a reasonable default"), with no dark mode, no theme switching, no per-component overrides. We're keeping the "no dark mode/switching shipped yet" part, but replacing the color itself and adding a typographic detail layer, both derived from real sites rather than picked off a hue wheel: warm paper background (`#F4EFEA`, MotherDuck's actual cream), ink (`#1d1f27`, PostHog's actual dark), a red-orange primary accent (`#f54e00`) and mustard secondary accent (`#f7a501`, both PostHog's), plus a "boxy detail" layer — monospace type on nav links/tag pills/footer quip/post titles, and a double-line footer rule — lifted from Latent Patterns' actual CSS conventions. templui's existing shape (0.65rem radius, 1px borders) is untouched; only color and this detail layer change. Every value stays a semantic CSS custom property, so a future dark variant is still a values-only addition, not a rewrite.

## Considered Options

Explored and rejected first: a single warm accent alone, a primary+secondary "pop" pairing, and a stark near-black/near-white "ink" base with a loud lime/magenta/blue/gold accent. All read as generic "AI-generated bold accent" choices once compared side by side. Grounding the palette in two named sites the author actually likes (PostHog, MotherDuck), verified against their real compiled CSS rather than recalled from memory, produced a more distinctive and defensible result. A pure Latent-Patterns-style boxy shape (sharp corners, thick borders throughout) was also considered and rejected in favor of keeping templui's existing rounded shape, with only the monospace/double-rule detail carried over.

## Consequences

Self-hosting JetBrains Mono (OFL-licensed, Latin-subsetted) becomes a build/embed dependency, following the same `ui/embed.go` static-asset pattern as `main.css`. It ships as one variable-font file covering the full `100 800` weight axis rather than two static 400/700 files — the templates that use it need `font-medium` (500) and `font-semibold` (600), neither of which is 400 or 700, so the range is load-bearing, not speculative. The monospace treatment now spans multiple templates (nav, tag, footer quip, post titles) rather than being contained to one component — future "chrome" text should deliberately decide whether it's boxy-detail (mono) or reading content (system-sans) rather than copying whichever neighbor happens to be closest. `CODING_STANDARDS.md:280` needs updating to match once this is implemented — it currently still describes the old single-blue-accent state.

The supporting neutral tones (`--secondary`/`--border`/`--input` at `#b0a89e`, `--muted`/`--accent` at `#e7e0d6`, `--muted-foreground` at `#7a7a72`) aren't sourced from PostHog or MotherDuck directly — they're warm tan/gray tones interpolated between the paper and ink endpoints, carried over from the same mockup iterations the author reviewed and approved during the grilling session, rather than freshly invented at implementation time.
